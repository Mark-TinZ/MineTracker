# Руководство по разработке (Contributing Guide)



## 🌿 Правила работы с Git (Git Flow)



Чтобы история коммитов оставалась чистой, а интеграция кода проходила без конфликтов, при работе на выделенной ветке придерживайтесь следующих правил:



1. **Синхронизация — превыше всего:** Перед началом работы над новой задачей **всегда** делайте `git fetch` и подтягивайте последние изменения из ветки `main`.

2. **Изоляция задач:** Для каждой новой задачи (новый хендлер, миграция БД, фиксы) создавайте отдельную ветку. 

   * Формат нейминга: `название-ветки/название-задачи` (например, `feature/server-stats`, `fix/rate-limit-bug` или `refactor/middlewares`).

3. **Атомарные коммиты:** Старайтесь коммитить изменения сразу по мере выполнения логического куска задачи. Не сваливайте десятки разных изменений в один огромный коммит.



## 📂 Структура проекта и Архитектура



Проект использует **Aiogram 3** и **SQLAlchemy**. Архитектура проекта жестко разделена на слои, чтобы отделить логику работы с Telegram от бизнес-логики и работы с БД.



* **Слой хендлеров (`mc_ping_bot/bot/`):** Здесь описываются только обработчики Telegram (message/callback), клавиатуры (keyboards), меню команд (commands_setup) и Middleware. **Правило:** В хендлерах не должно быть "тяжелой" бизнес-логики и прямых SQL-запросов (за исключением простых CRUD-операций через сессию). Хендлер должен получить данные, вызвать сервис и вернуть ответ пользователю.

* **Слой сервисов (`mc_ping_bot/services/`):** Здесь находится вся бизнес-логика. Например, класс `MinecraftService` инкапсулирует логику пингования серверов через `mcstatus`, а `RedisCacheManager` - логику кеширования. 

* **Слой базы данных (`mc_ping_bot/db/`):** Здесь находятся модели SQLAlchemy (`models.py`) и конфигурация подключения (`database.py`). 



## 🛠 Работа с базой данных (Alembic)



Мы используем **Alembic** для управления миграциями. Если вы изменили модели в `mc_ping_bot/db/models.py`:



1. **Обязательно** создайте новую миграцию:

   ```bash

   alembic revision --autogenerate -m "Краткое описание изменений, например: Add gamechat_thread_id to MonitoredServer"

   ```

2. Проверьте сгенерированный файл в папке `alembic/versions/` (Alembic иногда ошибается при автогенерации).

3. Примените миграцию к вашей локальной БД для проверки:

   ```bash

   alembic upgrade head

   ```



## 🧩 Dependency Injection (Внедрение зависимостей)



В проекте реализован механизм DI через `DatabaseMiddleware`. Это означает, что вам **не нужно** вручную создавать сессии БД внутри хендлеров.



* **Правильный подход:**

  Просто укажите аргумент `session: AsyncSession` или `mc_service: MinecraftService` в вашем хендлере, и Aiogram автоматически его туда передаст!

  ```python

  @router.message(Command("ping"))

  async def ping_handler(message: Message, session: AsyncSession, mc_service: MinecraftService):

      # Использование сессии или сервиса...

      pass

  ```



## 🎯 Приоритеты и задачи



1. **Разминка:** Начните с простых команд или рефакторинга текстов сообщений (`mc_ping_bot/bot/i18n.py`), чтобы набить руку на Aiogram 3.

2. **Работа со статусами серверов:** Если вы добавляете новый функционал для мониторинга (например, графики TPS), старайтесь изолировать логику сбора данных в фоновые воркеры (`mc_ping_bot/services/monitor.py`), чтобы не блокировать отправку сообщений в Telegram.

3. **Безопасность:** Если вы пишете административные команды, всегда проверяйте роль пользователя через базу данных, а не доверяйте Telegram ID. Для этого можно создать кастомный фильтр ролей.
